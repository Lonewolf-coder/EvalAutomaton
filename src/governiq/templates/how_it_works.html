How Evaluation Works

Overview

GovernIQ evaluates chatbot assessments using two independent pipelines that run against your bot. The results from both are merged into a single scorecard.

The evaluation engine is domain-agnostic. It doesn't know what a flight, appointment, or leave request is. It knows six testing patterns. A configuration file (called a "manifest") tells it what to check for each assessment type. This means new assessment types cost zero code changes.

The Two Pipelines

Pipeline A: Structural Analysis (CBM)

Parses your exported bot JSON and checks its structure:

  • Are all required dialogs present?
  • Does each dialog have the right nodes (entity, service, message, Agent Node)?
  • Are POST, GET, PUT, DELETE service nodes configured?
  • Are entity validation rules set up?
  • Are FAQs present with alternate phrasings?
  • Is DialogGPT enabled?

Key property: Deterministic. Same export always produces the same result. Can be re-run at any time.

Pipeline B: Live Conversation (Webhook)

An AI driver talks to your live bot and tests its behavior:

  • Can the bot greet users and show available options?
  • Can it collect all required entities for a booking?
  • Can it handle mid-conversation amendments (Agent Node test)?
  • Does data actually reach the Mock API?
  • Can it retrieve, modify, and cancel records?
  • Does it handle invalid input gracefully?

Key property: Tests real behavior. A bot can have the right structure but still fail in practice.

The Six Testing Patterns

Every task in every assessment maps to exactly one of these patterns:

PatternWhat the Engine DoesWhat It Checks
CREATEDrives a conversation to collect all required entities, then triggers a POST to your APIWere all entities collected? Was the record saved to the API correctly?
CREATE + AMENDSame as CREATE, but mid-conversation changes one entity value before confirmationDoes the Agent Node support amendment? Does the final record have the amended value?
RETRIEVEAsks the bot to look up an existing record using data from a previous taskDid the bot return the correct data? Does it match the API record?
MODIFYRetrieves a record then asks to change one or more fieldsWas the modification saved to the API? Do the new values match?
DELETEAsks the bot to cancel/delete a recordIs the record actually gone from the API?
EDGE CASEInjects an invalid value (wrong phone number, bad date, etc.)Does the bot respond with a clear error message? Does it offer to try again?

Scoring

CBM Structural — 40%

Does the bot have the right internal structure?

Webhook Functional — 40%

Does the bot actually work in conversation?

Compliance — 10%

Are platform settings correct? (DialogGPT, channels, language)

FAQs — 10%

Are FAQs implemented with correct answers and alternates?

PASS: 70%+

No critical failures

REVIEW: 50-69%

Some areas need work

FAIL: Below 50%

Major gaps found

State Seeding (Fairness)

If the bot fails to create a booking through conversation (Task 2), Tasks 3 and 4 (retrieval, modification) cannot be tested fairly. To prevent unfair cascade failures, the engine will:

  1. Validate the candidate's Mock API schema against what the bot tried to send
  2. If the schema matches, insert a synthetic record into the Mock API
  3. Run Tasks 3 and 4 against the seeded data
  4. Flag the scorecard with an amber warning: "Tasks tested against seeded data"

This ensures you still get feedback on retrieval and modification, even if booking failed.

Key Decisions

1
DialogGPT: Enabled = Pass
The assessment requires DialogGPT to be enabled. If it's ON in your bot, this passes. If OFF, this is a critical failure.
2
Both pipelines are independent
CBM and webhook pipelines run separately. They don't affect each other during execution. Scores are merged only at the end.
3
Booking 2 = Amendment test only
Booking 2 is not a separate booking. It's the same flow as Booking 1, but mid-conversation one entity value is changed to test the Agent Node's amendment capability.
4
Tasks 3 & 4 reuse Task 2 data
The evaluation is stateful. Data from the booking task is used for retrieval and modification. This mirrors how a real user would interact with the bot.
